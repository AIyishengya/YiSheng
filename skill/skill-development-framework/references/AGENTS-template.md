# 项目开发约束

> 此文件为跨工具通用指令，被 Claude Code、OpenAI Codex、Cursor、GitHub Copilot、Gemini CLI、Windsurf、Cline、Aider 等 20+ AI 编码工具识别。

---

## 语言要求

- 所有 skill/指令文档、代码注释、说明文字必须使用**中文**
- 代码本身保持英文（编程语言的自然语言），注释使用中文
- 技术术语首次出现时需给出中文解释

---

## Skill 开发规范

### 核心原则：知识与流程分离

- **Skill 提供知识**：领域模式、原则、代码示例、验证清单、反模式
- **Agent 提供流程**：执行步骤、工具调用序列、决策逻辑、错误恢复

禁止在 skill 中内嵌具体的工具调用指令或执行流程。

### 工具中立

- 不指定具体工具名称（不写"使用 Bash 运行 X"、"使用 Grep 搜索 Y"）
- 描述所需的**能力**，让 agent 根据自身工具集选择实现方式
- 对需要写权限的操作，使用条件语句："如果你具有写入权限……否则报告……"

### 模型无关

- 禁止使用模型特定提示词：`think harder`、`ultrathink`、`think step by step`、`深度思考`、`一步步推理`
- 改为描述所需的分析深度："将每个问题与 X 进行逐一对照评估"

### Skill 结构要求

每个 skill 须包含以下节：

1. **触发条件**（何时激活）——具体场景，非抽象描述
2. **核心原则**——带正反对比的代码示例（✅ 正确 / ❌ 错误）
3. **检查清单**——工具中立的可验证条目
4. **常见陷阱**——症状 / 原因 / 修复 三段式
5. **参考资料**——相关 skill 的显式路径引用

### 代码示例

- 所有代码示例必须标注编程语言
- 语言无关的原则，提供多语言示例
- 代码注释使用中文

### 交叉引用

- 使用显式路径格式引用其他 skill
- 禁止循环依赖
- 依赖其他 skill 时需显式声明

---

## 编码规范

### 不可变性（关键）

始终创建新对象，永远不修改现有对象。防止隐藏副作用，简化调试，支持安全并发。

### 文件组织

多个小文件优于少数大文件：
- 高内聚、低耦合
- 典型 200-400 行，上限 800 行
- 按功能/领域组织，不按类型

### 错误处理

- 在每一层显式处理错误
- UI 层提供用户友好的错误消息
- 服务端记录详细的错误上下文
- 永远不静默吞掉错误

### 输入验证

在系统边界处验证所有输入：
- 使用 schema 验证
- 快速失败，提供清晰错误消息
- 永远不信任外部数据

### 提交规范

```
<type>: <描述>

<可选正文>
```

类型：feat、fix、refactor、docs、test、chore、perf、ci

---

## 安全规范

提交前必查：
- [ ] 无硬编码的密钥（API 密钥、密码、令牌）
- [ ] 所有用户输入已验证
- [ ] SQL 注入防护（参数化查询）
- [ ] XSS 防护（HTML 净化）
- [ ] 错误消息不泄露敏感数据
- [ ] 密钥通过环境变量或密钥管理器存储

---

## 测试要求

最低覆盖率：80%

测试类型：
- **单元测试**——函数、工具函数、组件
- **集成测试**——API 端点、数据库操作
- **E2E 测试**——关键用户流程

工作流：写测试（红）→ 实现（绿）→ 重构（改进）→ 验证覆盖率

---

## 反模式（禁止项）

- ❌ 在 skill 中内嵌工具调用流程
- ❌ 指定具体工具名称
- ❌ 使用模型特定提示词
- ❌ Skill 与 agent 内置流程冲突
- ❌ 超过 5000 token 的单体 skill（拆分到子文件）
- ❌ 无交叉引用的孤儿 skill
- ❌ 代码示例未标注语言
- ❌ 假设 agent 具有写权限
